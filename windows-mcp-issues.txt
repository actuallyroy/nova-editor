8. [HIGH] start_sandbox is destructive when a sandbox is already running
----------------------------------------------------------------------
Observed (reported by user, consistent with my repeated failures):
  - Calling start_sandbox while a sandbox is ALREADY active pops Windows Sandbox's
    native "Only one running instance of Windows Sandbox is allowed" dialog.
  - Clicking OK on that dialog CLOSES the existing sandbox but does NOT spawn the
    new (re-mounted) one. Net result: you lose the running sandbox and get nothing.
  - This is why every attempt to remount E:\Nova Editor over an already-running
    (windows-mcp-mounted) sandbox has been destructive, and why reconnect+remount
    has been so painful all session.
Also seen: a launch_app call returned "sandbox helper disconnected (read ECONNRESET)"
  followed by ping -> "connect ETIMEDOUT 9335", i.e. the helper died mid-session even
  though sandbox_status still reported active (stale status). The force-kill removal
  noted under #6 did not prevent an ECONNRESET recurrence here.
Suggested fix:
  - start_sandbox should detect an already-running sandbox and EITHER reuse/adopt it
    (preferred), OR gracefully shut it down FIRST and wait for full teardown before
    spawning the replacement — never leave the user with zero sandboxes.

RESOLUTION (windows-mcp dev, 2026-05-28):
  8 FIXED (non-destructive). start_sandbox now:
      - adopts + REUSES an already-running sandbox (via the ready.json scan) and returns
        its status — the common case (same/no project_dir) never spawns or closes anything.
      - if a DIFFERENT project_dir is requested while one is running, returns a clear error
        telling you to close the sandbox window first. It NEVER force-closes (force-kill
        wedges Windows Sandbox until reboot) and NEVER triggers the "only one instance"
        dialog (it doesn't attempt a 2nd spawn while one is up). So you can't end up with
        zero sandboxes. (sandbox.ts: startSandbox)
      - the destructive "OK on the dialog closes the old + spawns nothing" path is gone:
        the old force-kill recycle was removed entirely earlier (issue #6).
    Stale-status part: sandbox_status now PINGS the helper before reporting active; a dead
    helper (closed sandbox / ECONNRESET) is reported inactive and the tracking is cleared,
    so the next call cleanly re-adopts or re-spawns. (sandbox.ts: statusVerified)
    NOTE: like the others, needs the fresh Node server (full Claude restart) to be live.
  - Have sandbox_status actually probe the helper (TCP) rather than report cached
    "active:true" after the helper has died.

1. [HIGH] press_key does not deliver a translated character (no WM_CHAR) for printable keys
----------------------------------------------------------------------
Observed:
  - press_key {key:"DOWN"} and other NAMED keys work — Nova's cursor moved, status bar
    updated Ln 1 -> Ln 2. (These are read from WM_KEYDOWN / virtual-key, so they arrive.)
  - press_key for letter keys H,E,L,L,O produced NO text insertion. Status stayed "Ln 2, Col 1",
    line stayed empty. Each call returned {ok:true}.
Likely cause:
  - press_key posts WM_KEYDOWN/WM_KEYUP for the VK code but never TranslateMessage()/WM_CHAR,
    so apps that read the TRANSLATED character (the standard text-input path: winit's
    KeyEvent.text, which is sourced from WM_CHAR) receive nothing.
Suggested fix:
  - After the key-down, synthesize the corresponding WM_CHAR (or use the SendInput scan-code
    path that lets the OS translate it), so text-input-driven apps get the character.
Impact: cannot type into any app that consumes WM_CHAR rather than raw VK (most text editors,
  game/GPU UIs, anything using winit/SDL/GLFW text input).

----------------------------------------------------------------------
2. [HIGH] type_text silently no-ops on windows with no UIA text element
----------------------------------------------------------------------
Observed:
  - type_text {text:"..."} returned {ok:true} but inserted nothing into Nova's editor.
Likely cause:
  - type_text uses a UIA ValuePattern.SetValue (or similar) against a focused accessibility
    element. Nova is fully custom GPU-rendered and exposes no UIA text control, so the call
    finds no target and does nothing — yet still reports success.
Suggested fix:
  - When no UIA text target is found, fall back to SendInput with KEYEVENTF_UNICODE
    (synthesizing WM_CHAR per character).
  - And/or return ok:false (or a "no_uia_target" note) instead of ok:true so callers know
    nothing happened.

----------------------------------------------------------------------
3. [MED] screenshot "The handle is invalid" — intermittent (~50%)
----------------------------------------------------------------------
Observed:
  - screenshot {target:"sandbox"} (full desktop) failed repeatedly with "The handle is invalid",
    especially right after sandbox boot, and intermittently afterward.
  - screenshot {target:"sandbox", pid:<nova>} (window-targeted) was far more reliable.
Likely cause:
  - Desktop Duplication / GDI capture handle going stale or not (re)acquired for the session;
    the per-window path uses a different capture handle that survives.
Suggested fix:
  - Recreate the duplication/output handle on "invalid handle" and retry once before erroring;
    or warm up the capture after sandbox boot.

----------------------------------------------------------------------
4. [MED] launch_app reports "could not launch app" even when the app launched fine
----------------------------------------------------------------------
Observed:
  - launch_app for nova.exe returned the error string "could not launch app", but list_apps
    immediately showed the process running with a valid, active window titled "Nova".
  - This was a consistent false-negative for this GUI app.
Likely cause:
  - Success is gated on detecting a top-level window within a short window-wait; winit/wgpu apps
    create their window a beat later (adapter/device init), so the heuristic times out and
    reports failure even though launch succeeded.
Suggested fix:
  - Return the spawned pid regardless and let the caller verify via list_apps; or lengthen /
    poll the window-wait; or treat "process alive" as success even if no window yet.

5. [MED] run_script rejects fully-qualified MCP tool names in step.tool
----------------------------------------------------------------------
Observed:
  - {"tool":"mcp__windows__press_key", ...} -> error "unknown tool" (stopped_at:0).
  - {"tool":"press_key", ...} -> works.
Note: every other surface in this MCP is addressed with the fully-qualified

Suggested fix:
  - Accept both the short name and the "mcp__windows__"-prefixed name; or state the requirement

9. [MED] no mouse click-drag primitive (press-move-release)
----------------------------------------------------------------------
Date: 2026-05-28 (Nova session)
Observed:
  - The MCP exposes hover {x,y} (move, no button), click {x,y} (press+release in
    place), try_click, scroll, press_key — but NO way to press the left button,
    move while held, and release. i.e. no drag gesture.
Impact:
  - Cannot test any drag-driven UI: resizing panels (just added a draggable sidebar
    splitter to Nova and could not exercise it), text selection by dragging in the
    editor, drag-to-reorder tabs, scrollbar thumb dragging, window-edge resize, etc.
    These are core editor interactions, so this blocks a whole class of verification.
Suggested fix:
  - Add a drag tool, e.g. drag {from:{x,y}, to:{x,y}, button?, steps?} that does
    SendInput LEFTDOWN at `from`, several MOUSEMOVE steps to `to`, then LEFTUP —
    with a few intermediate moves so apps that track WM_MOUSEMOVE deltas (winit
    CursorMoved) see a continuous drag, not a teleport.
  - Optionally expose raw mouse_down {x,y,button} / mouse_up {x,y,button} so callers
    can compose arbitrary press-hold-move-release sequences.
